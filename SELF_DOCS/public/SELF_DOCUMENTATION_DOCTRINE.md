# SELF Documentation Doctrine

## Immutable Principles for Documentation Excellence

**Effective Date: Saturday December 20 2025**
**Governance: Documentation Council**
**Amendment Process: Unanimous Council Approval**

---

## Core Principles

### 1. User-Centered Design
**All documentation serves end users first.** Technical audiences receive specialized paths, but the foundation serves those who need SELF most. Documentation exists to empower, not impress.

### 2. Safety as Documentation Foundation
**Safety documentation is immutable.** User safety documents receive highest protection. Changes require extraordinary justification and follow strict governance protocols.

### 3. Progressive Disclosure
**Start simple, enable depth.** Essential information is immediately accessible. Advanced details await those who seek them. No user should be overwhelmed by information they don't need.

### 4. Evidence-Based Content
**Best practices grounded in research.** Recommendations supported by user data, expert consultation, and empirical evidence. Documentation evolves with knowledge.

### 5. Concise and Actionable
**Every word serves a purpose.** Eliminate redundancy, prioritize clarity, focus on outcomes. Documentation that cannot be understood in one sitting requires revision.

### 6. Audience Segmentation
**One size does not fit all.** Researchers, executives, developers, and users receive documentation tailored to their needs and mental models. Cross-references enable exploration.

### 7. Continuous Evolution with Stability
**Documentation improves but principles endure.** Content updates regularly, but core doctrines remain stable. Changes serve users, not preferences.

## Operational Guidelines

### Documentation Creation
- **Start with user needs**: What must users know to succeed?
- **Write for comprehension**: Use plain language, avoid jargon
- **Test for clarity**: Can a new user understand and act?
- **Maintain consistency**: Use established patterns and terminology

### Documentation Maintenance
- **Regular review cycles**: Quarterly updates for user-facing content
- **User feedback integration**: Real usage informs improvements
- **Version control discipline**: Changes tracked, reversible
- **Quality gates**: No document published without peer review

### Documentation Governance
- **Safety documents**: Require council approval for changes
- **Breaking changes**: Impact assessment mandatory
- **Deprecation process**: Clear migration paths for major updates
- **Accessibility standards**: WCAG compliance for all user content

## Documentation Standards

### Structure Requirements
- **Clear hierarchy**: H1 → content flow → actionable conclusions
- **Scannable format**: Bullet points, bold key terms, white space
- **Progressive complexity**: Simple concepts first, complexity later
- **Action orientation**: Every section ends with clear next steps

### Content Standards
- **Empathy first**: Consider user's emotional state and context
- **Inclusive language**: Welcome all backgrounds and experiences
- **Error prevention**: Anticipate user mistakes and confusion
- **Help integration**: Easy paths to human assistance when needed

### Quality Standards
- **Accuracy**: Information verified, up-to-date, correct
- **Completeness**: Covers essential user journeys and edge cases
- **Usability**: Tested with real users, refined based on feedback
- **Maintainability**: Easy to update, clear ownership, documented processes
